## Problem
Service `tkrclass` is a REPO service built with RAILPACK (provider: node, npm). The build fails at the BUILD_IMAGE stage during the install step:
```
npm error Exit handler never called!
npm error This is an error with npm itself.
Build Failed: process "npm ci" did not complete successfully: exit code: 1
```
SNAPSHOT_CODE succeeds; BUILD_IMAGE fails on `npm ci` after ~90 seconds. There is no Railway config file; all config is dashboard/default. A `nixpacks.toml` exists in the repo but is ignored because Railway uses Railpack, not Nixpacks.

## Diagnosis
`package-lock.json` was generated inside Replit's environment behind its "package firewall". Many dependency entries have `resolved` URLs pointing to `http://package-firewall.replit.local/npm/...` (e.g. `@img/sharp-linux-x64`, `@img/colour`, `@emnapi/runtime`, `append-field`, `buffer-from`). `npm ci` installs strictly from the lockfile and fetches these exact URLs. On Railway's build network, `package-firewall.replit.local` does not resolve, so npm stalls on the fetch and crashes with its internal "Exit handler never called!" bug. The last successful deploy predates these dependencies; the failure started when the lockfile with Replit-internal URLs was committed.

## Fix
This fix cannot be completed by editing text files alone because a valid lockfile must be regenerated by npm. Steps for the user to perform locally:
1. Delete `package-lock.json` at the repo root.
2. Delete `nixpacks.toml` at the repo root (unused by Railway/Railpack; it references `npm install --omit=dev` which will not take effect).
3. Run `npm install` on a machine with normal access to `https://registry.npmjs.org`. Verify the regenerated `package-lock.json` contains only `registry.npmjs.org` resolved URLs and no `package-firewall.replit.local` entries: `grep replit.local package-lock.json` should return nothing.
4. Commit and push both changes to `main`.
This resolves the issue because `npm ci` will then fetch all packages from the public npm registry, which is reachable from Railway's builders. If regenerating the lockfile locally is not feasible immediately, an interim option is to delete `package-lock.json` from the repo so Railpack falls back to `npm install` (which resolves fresh against the public registry) instead of `npm ci`.
